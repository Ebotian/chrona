自顶向下(先设计后实现)的逻辑在运用ai时仍然尤为管用.

设计完原理图可以导出网表经过程序处理(例如`gen_pinmap_md.py`)生成引脚映射表,然后在`src/main.h`中定义引脚的宏,例如:

```c
#define LED_PIN GPIO_PIN_5
```
然后在`src/main.c`中使用这些宏定义来控制引脚:

```c
#include "main.h"
void setup() {
    // 初始化LED引脚
    GPIO_InitTypeDef GPIO_InitStruct = {0};
    __HAL_RCC_GPIOA_CLK_ENABLE();
    GPIO_InitStruct.Pin = LED_PIN;
    GPIO_InitStruct.Mode = GPIO_MODE_OUTPUT_PP;
    GPIO_InitStruct.Pull = GPIO_NOPULL;
    GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_LOW;
    HAL_GPIO_Init(GPIOA, &GPIO_InitStruct);
}
void loop() {
    // 切换LED状态
    HAL_GPIO_TogglePin(GPIOA, LED_PIN);
    HAL_Delay(500); // 延时500毫秒
}
```
这样就可以在代码中使用宏定义来控制引脚了,而不是直接使用具体的引脚号,这使得代码更易于维护和移植.也便于自动化生成.

读数据手册依然很重要.可以用`pdftotext`将可选文本pdf(不可选的要先ocr)转为文本交给ai分析,然后根据特性设计相关程序.当然一些图片,比如三极管设计的增益曲线图还可能需要手动分析才能确保准确,或者是人工提取图像的特征点信息进行描述,比如给定大致的放大倍数.

如果在一个ai上迟迟没有进展,适时更换ai可以有更好的效果.目前来看,gemini比较死板,gpt4.1精简,o4-mini比较迟缓但聪明.不要用claude,它经常给出太多我没有要求的功能导致代码过于冗长从而失去控制,如果有这种情况发生,就意味着要清零再来.
